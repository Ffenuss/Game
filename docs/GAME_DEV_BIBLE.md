# Библия разработки игры

Документ предназначен как постоянный внутренний стандарт проекта `Пепельный Дол: Клеймо Пустоши`.
Он важнее кратких заметок, разовых обсуждений и импровизаций в отдельных задачах.

## 1. Назначение документа

Этот документ описывает, как проект должен создаваться, поддерживаться, тестироваться, расширяться и подготавливаться к выпуску.

Он нужен для того, чтобы:

- будущие Codex-задачи не расходились по архитектуре;
- замены ассетов не ломали сцены и скрипты;
- Android-first ограничения не игнорировались ради удобства редактора;
- лицензии оставались проверяемыми;
- каждый шаг можно было остановить, проверить и откатить;
- визуальный стиль и игровые правила не превращались в случайный набор решений.

Если новая задача противоречит этой книге, сначала меняют документированное правило, а не проект “по-тихому”.

## 2. Как Codex обязан использовать этот документ

Перед редактированием проекта Codex обязан:

1. Прочитать `AGENTS.md`.
2. Прочитать `docs/GAME_DEV_BIBLE.md`.
3. Прочитать `docs/ASSET_SOURCES_2026.md`.
4. Прочитать `docs/LICENSE_POLICY.md`.
5. Прочитать `docs/GODOT_46_PROJECT_RULES.md`.
6. Прочитать `docs/ANDROID_GAME_RULES.md`.
7. Прочитать `PLANS.md`.

Правила работы:

- не выдумывать успех, если проверка не запускалась;
- не делать большие незапланированные переписывания;
- не ломать существующие рабочие сцены ради “идеальной” архитектуры;
- обновлять документы после завершённой фазы;
- обновлять `data/assets/asset_manifest.json` при любом изменении ассетов;
- фиксировать blocker, если задача упирается в среду, права, лицензию или редактор;
- делать маленькие безопасные коммиты;
- останавливать работу перед разрушительными изменениями;
- сохранять русский visible UI текст;
- сохранять Android-first приоритет.

Если задача кажется простой, но затрагивает архитектуру, ассеты или лицензии, Codex обязан всё равно свериться с этой книгой.

## 3. Продуктовое видение игры

**Название:** `Пепельный Дол: Клеймо Пустоши`

**Жанр:** Android-first dark-fantasy pixel-art action RPG.

**Платформа в приоритете:** Android, landscape, touch-first.

**Позднее:** desktop, затем возможно iOS.

**Тон:** холодная долина, пепел, каменный лес, старая штольня, редкая тёплая печная подсветка, читаемый ближний бой, компактные атмосферные локации.

**Цель вертикального среза:** не огромная RPG, а маленький, законченный и показываемый игровой кусок, который:

- запускается стабильно;
- играетcя на Android;
- содержит понятный боевой цикл;
- имеет один квест;
- имеет один чекпойнт и один слот сохранения;
- выглядит как игра, а не как редакторский тест.

**Что нужно строить сейчас:**

- небольшой playable loop;
- компактный мир;
- один основной маршрут;
- минимум систем, необходимых для удовольствия и проверки.

**Что нельзя строить сейчас:**

- десятки квестов;
- длинную сюжетную ветку;
- сложный боевой мета-слой;
- огромный мир;
- сетевые механики;
- сложный диалоговый движок;
- систему инвентаря уровня коммерческой RPG;
- лишние подсистемы “на будущее”, если они не участвуют в текущем срезе.

## 4. Главные правила разработки игр

### 4.1 Маленький рабочий цикл важнее большой незавершённой системы

Нужно сначала получить:

- запуск;
- движение;
- взаимодействие;
- урон;
- чекпойнт;
- сохранение;
- журнал/квест;
- читаемость на Android.

Только потом расширять объём.

### 4.2 Игровая обратная связь важнее количества контента

Игрок должен понимать:

- куда идти;
- что опасно;
- что можно нажать;
- когда он получил урон;
- когда стамина закончилась;
- где чекпойнт;
- когда начался квест;
- что открылась диалоговая/меню поверхность.

Если есть выбор между “ещё один пустой NPC” и “лучшее визуальное понимание боя”, выбирается понимание боя.

### 4.3 Одна законченная локация лучше пяти пустых

В текущем срезе лучше:

- одна компактная, но читаемая локация;
- один маршрут;
- одна маленькая зона-переход;
- несколько сильных визуальных ориентиров;

чем большая карта без глубины.

### 4.4 Читаемость важнее декора

Для этой игры приоритеты такие:

1. понятный силуэт;
2. понятный путь;
3. понятная точка взаимодействия;
4. понятный HUD;
5. атмосфера;
6. орнамент;
7. мелкий декор.

Если декор мешает читать бой, декор убирается.

### 4.5 Mobile UX надо проверять на устройстве

Эмулятор и окно редактора не дают полного ответа.

Обязательно:

- проверять landscape composition;
- проверять reachability больших пальцев;
- проверять, не закрывает ли HUD важные объекты;
- проверять читаемость Russian labels;
- проверять, не слишком ли много прозрачных оверлеев;
- проверять frame pacing хотя бы на одном реальном устройстве.

### 4.6 Системы должны быть модульными

Нельзя складывать всё в один “game manager”.

Нужно использовать:

- `HealthComponent`
- `StaminaComponent`
- `HitboxComponent`
- `HurtboxComponent`
- `InteractionComponent`
- `CheckpointComponent`
- `QuestTriggerComponent`
- `DialogueTriggerComponent`

Через сигналы и локальные компоненты проще чинить, заменять и тестировать.

### 4.7 Ассеты должны заменяться без переписывания игры

Путь к каждому ассету должен быть:

- стабильным;
- задокументированным;
- проверяемым через manifest;
- пригодным для замены без массовой правки кода.

### 4.8 Сохранение должно быть устойчивым

Сохранение не может падать из-за:

- отсутствующего файла;
- старой версии формата;
- частично битых данных;
- отсутствующего ассета;
- изменения сцены без миграции.

Если save повреждён, проект должен продолжать работать на дефолтных значениях.

### 4.9 Если нельзя протестировать, это не done

Нельзя считать фичу завершённой, если:

- нет headless validation;
- нет static validation;
- нет manual Android checklist;
- нет хотя бы одного осмысленного игрового теста.

## 5. Scope discipline

### 5.1 Вертикальный срез

Вертикальный срез — это небольшой, но законченный опыт:

- начало;
- понимание;
- действие;
- обратная связь;
- завершение.

### 5.2 Прототип

Прототип доказывает возможность.
Он может быть грубым, но не должен быть хаотичным.

### 5.3 Release candidate

Release candidate для этой игры — это не “всё готово”, а:

- стабильно запускается;
- playable loop не ломается;
- UI читаем;
- сохранение не опасно;
- лицензии прозрачны;
- документация актуальна.

### 5.4 Как решать, что делать сейчас

Фича входит в текущий объём, если она:

- усиливает playable loop;
- помогает тестировать;
- нужна для Android-управления;
- нужна для чтения мира;
- нужна для сохранения или диалога;
- уменьшает технический риск.

Фича откладывается, если она:

- не участвует в текущем loop;
- делает архитектуру тяжелее;
- добавляет лицензный риск;
- требует огромного контента;
- не может быть протестирована сейчас.

### 5.5 Definition of done

Фича считается done только если:

- работает;
- не ломает существующее;
- описана в документации;
- прошла доступную валидацию;
- ясно как её заменить или расширить.

## 6. Godot 4.6 standards

### 6.1 Рендер и платформа

- Использовать Godot 4.6.
- Использовать Compatibility renderer.
- Держать проект в landscape orientation.
- Сохранять pixel-art sharpness через nearest filtering и pixel snapping.

### 6.2 Сцены и узлы

- Предпочитать отдельные reusable scenes.
- Не создавать огромные сцены, если модуль можно вынести.
- Не привязывать unrelated systems к hard-coded path-цепочкам.
- Использовать `CanvasLayer` для HUD и поверхностных UI-слоёв.

### 6.3 Ввод

- Все действия должны быть в `Input Map`.
- Нужен touch-first UI и keyboard fallback.
- Имя action должно быть стабильным.
- Нельзя хардкодить ввод только в один device mode.

### 6.4 Скрипты

- GDScript only.
- Typed GDScript wherever practical.
- Один скрипт — одна ответственность.
- Минимизировать глобальные side effects.
- Использовать сигналы для слабой связанности.

### 6.5 TileMapLayer

- Использовать `TileMapLayer`, не deprecated `TileMap`.
- Карта должна быть редактируемой.
- Слои должны быть ясны по функции.
- Коллизии, переходы и декор должны быть разделены.

### 6.6 Resources и JSON

- Данные квестов, диалогов, локаций и ассетов хранить отдельно от логики, когда это практично.
- Использовать локальные JSON или Resource файлы.
- Для редактируемого контента не смешивать текст и механику без причины.

### 6.7 Autoloads

Рекомендуемые автолоады:

- `GameState`
- `SaveManager`
- `AssetCatalog`
- `QuestManager`
- `SceneRouter`

Правило: автолоад остаётся маленьким. Если в нём становится слишком много обязанностей, это признак, что нужен отдельный компонент или ресурс.

### 6.8 Сохранение

- Только локальные save files.
- Только один рабочий слот на старте.
- Корруптированный файл не должен падать.

### 6.9 Экспорт Android

- Экспорт не считается завершённым без проверки конфигурации.
- `.godot/`, APK, AAB и keystore не коммитятся.
- Landscape и Compatibility должны быть сохранены в экспортном контуре.

## 7. Architecture rules

### 7.1 Принципы

- Composition over inheritance там, где это помогает.
- Повторно используемые вещи выносить в компоненты.
- Разделять “игровую сущность” и “ассет”.
- Не смешивать UI-логику с боевой логикой.
- Не делать scene router, save manager или quest manager универсальной свалкой.

### 7.2 Asset path discipline

- Не рассыпать raw asset paths по скриптам.
- Использовать `AssetCatalog`.
- Стабильные runtime paths сохранять при замене.
- Никогда не ссылаться на `third_party/` из сцены.

### 7.3 Data flow

- Сцена получает данные, а не “знает всё”.
- Данные живут в `data/`.
- Ресурсы живут в `resources/`.
- Runtime assets живут в `assets/runtime/`.
- Генератор placeholder-ассетов живёт в `assets/generated/placeholders/`.

### 7.4 Recommended scene mental model

**Player**

- movement
- combat
- health/stamina
- interaction
- camera

**Enemy**

- lightweight state
- detection
- attack telegraph
- damage response

**Location**

- layered tiles
- spawn points
- interactables
- atmosphere
- camera framing

**HUD**

- compact controls
- health/stamina
- prompt surfaces
- pause/menu surfaces

## 8. Android-first rules

- Жизненно важен landscape composition.
- HUD должен оставлять центр экрана свободным.
- Джойстик должен сидеть слева снизу.
- Основные кнопки должны быть справа снизу.
- Пауза и вспомогательные поверхности должны быть компактными.
- Русский visible UI текст должен быть коротким, читабельным и однозначным.
- Все крупные overlays должны учитывать safe zones.
- Тёмный фон и лёгкий vignette лучше, чем серый пустой экран.
- Performance budget должен быть рассчитан на mid-range Android.
- Избыточная прозрачность и дорогие эффекты вредят сильнее, чем кажется в редакторе.

## 9. Pixel-art rules

- База тайла: `16x16`.
- Игрок: примерно `32x48`.
- UI icons: `16x16`, `24x24`, `32x32`.
- Нельзя допускать blurry scaling.
- Нужны ясные силуэты.
- Нужен контраст между ground / wall / interactable / enemy / player.
- Важна единая палитра:
  - dark graphite;
  - muted stone gray;
  - cold blue-gray;
  - faded brown;
  - pale ash;
  - warm furnace orange;
  - restrained dark red;
  - muted green and blue.

### Чего избегать

- слишком ярких кислотных цветов;
- постоянного белого UI;
- повторяющихся tile-рисунков без шумовой вариации;
- неразборчивых миниатюр;
- смешения резко разных художественных стилей.

## 10. Level design rules

- Каждая карта должна быть компактной и читаемой.
- Путь должен быть очевиден.
- Стены, обрывы и вода должны визуально различаться.
- Переходы должны читаться как переходы.
- Чекпойнт должен быть виден.
- Враги должны появляться там, где их можно понять и обойти.
- Камера не должна показывать пустоту больше, чем нужно.
- Rectangular test room выглядит допустимо только как временный production scaffold.

### Обязательные слои

- `GroundLayer`
- `GroundDetailLayer`
- `RoadLayer`
- `WaterLayer`
- `WallLayer`
- `BackDecorationLayer`
- `FrontDecorationLayer`
- `CharactersLayer`
- `EffectsLayer`
- `FogLayer`
- `UILayer`

## 11. Combat design rules

- Светлая атака должна иметь стамина-стоимость и окно hitbox.
- Рывок должен иметь стамина-стоимость и короткую неуязвимость.
- Удар врага должен читаться телеграфом.
- Получение урона должно иметь feedback.
- Смерть не должна ломать сцену.
- Respawn должен возвращать на checkpoint.
- Combat должен быть понятен на touch controls без сложной комбинации кнопок.

### Балансный принцип

Для вертикального среза лучше:

- меньше умений;
- сильнее читаемость;
- выше цена ошибок;
- яснее feedback.

## 12. UI/UX rules

- Visible labels in Russian.
- Кнопки должны быть под thumbs.
- На экране не должно быть постоянного debug-looking overlay.
- Меню паузы, журнал, инвентарь и диалог не должны перекрывать основной бой без причины.
- Плотность текста должна быть низкой.
- Opacity должна быть достаточной для игры под UI.
- Pressed state должен быть виден.
- Важный prompt должен появляться только когда он нужен.

### Что хорошо выглядит

- компактная панель здоровья/стамины;
- маленький, но заметный pause;
- action cluster справа снизу;
- короткие подписи;
- panel chrome, похожий на игровую UI.

### Что плохо выглядит

- большие белые слова поверх карты;
- кнопки в центре поля;
- блоки, похожие на инспектор Godot;
- одинаковые серые квадраты по всем краям экрана.

## 13. Save/load rules

- Один локальный save slot на старте.
- Автосейв после checkpoint activation.
- Данные должны включать:
  - позицию игрока;
  - location id;
  - checkpoint id;
  - health;
  - stamina;
  - inventory scaffold;
  - quest scaffold;
  - settings.
- Битый save файл не должен приводить к падению.

### Обработка ошибок

- Если save отсутствует, игра стартует с дефолтов.
- Если формат изменился, нужна миграция или безопасный fallback.
- Если чтение не удалось, выводится лог, а игра продолжает жить.

## 14. Asset pipeline rules

Pipeline:

`third_party/assets/`
↓
`tools/normalize_assets.gd`
↓
`assets/runtime/`
↓
`scenes / scripts / resources`

### Правила

- Runtime path должен быть каноническим.
- Source asset копия должна сохраняться только если она нужна для audit.
- Каждый third-party файл должен иметь proof.
- Manifest должен знать, что это за asset, откуда он пришёл и чем он заменяем.
- Placeholder assets остаются fallback, пока их не заменили официально.

### Нельзя

- ссылаться на `third_party/` из gameplay scenes;
- импортировать “что-нибудь бесплатное” без проверки;
- использовать assets без stable source page;
- терять license proof;
- считать картинку safe только потому, что она лежит в интернете.

## 15. Licensing rules

См. также `docs/LICENSE_POLICY.md`.

Короткое правило:

- безопасно автоматически только то, что явно CC0 / Public Domain / оригинально создано;
- всё остальное требует review;
- неизвестное лицензирование запрещено;
- “free” не означает “safe”.

### Обязательно архивировать

- source page;
- license name;
- creator;
- download date;
- runtime path;
- hashes;
- usage locations.

## 16. QA rules

### Статические проверки

- структура репозитория;
- наличие нужных папок и файлов;
- manifest парсится;
- no third-party direct references;
- no deprecated `TileMap`;
- no C#;
- no native plugins;
- no APK/AAB/keystore.

### Runtime checks

- headless startup smoke;
- scene parse;
- asset resolve;
- load main scene;
- отсутствие крэша при отсутствии декоративного ассета.

### Android checks

- landscape;
- touch movement;
- joystick reach;
- combat buttons;
- checkpoint;
- dialogue;
- journal;
- inventory;
- pause;
- screenshot readability.

## 17. Git rules

- inspect `git status` before work;
- use focused commits;
- не коммитить `.godot/`;
- не коммитить APK/AAB/keystores;
- не переписывать историю без причины;
- не скрывать незавершённые изменения;
- не делать destructive rollback ради “чистоты”.

## 18. What to do / what not to do

### Делать

- делать маленькие законченные шаги;
- использовать русские тексты UI;
- обновлять docs после изменений;
- проверять asset manifest после замены ассетов;
- держать gameplay и presentation разделёнными;
- сохранять плавающую заменяемость ассетов;
- тестировать Android-first layout на device-sized viewport.

### Не делать

- раздувать одну фичу в полсотни подсистем;
- писать огромный autoload;
- добавлять ассеты с unclear license;
- прятать проблемы за “потом поправим”;
- менять архитектуру без документации;
- оставлять пустые карты, которые читаются как редакторский тест;
- считать задачу завершённой без валидации.

## 19. Практические шаблоны для будущих задач

### Шаблон обычной Codex-задачи

1. Прочитать обязательные документы.
2. Сказать, что именно будет изменено.
3. Проверить текущий статус и связанные файлы.
4. Внести одну фокусную правку.
5. Прогнать доступную валидацию.
6. Обновить `PLANS.md` и релевантные docs.

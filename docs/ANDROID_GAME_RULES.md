# Правила Android-first игры

Этот документ фиксирует правила компоновки и поведения игры на Android для `Пепельный Дол: Клеймо Пустоши`.

## 1. Screen composition

- Игра проектируется под landscape.
- Центр экрана должен оставаться по возможности свободным во время обычного движения.
- HUD не должен выглядеть как desktop overlay, перенесённый на телефон.

## 2. Touch controls

- Джойстик слева снизу.
- Основные действия справа снизу.
- Пауза компактно справа сверху.
- Кнопки должны иметь понятный pressed state.
- Touch controls должны быть отдельной системой, а не “экранными кнопками ради галочки”.

## 3. Safe zones

- Не размещать важные элементы в зонах, где их перекрывают жесты системы.
- Не приближать важные кнопки вплотную к кромкам без причины.
- Держать внутренние отступы на разных aspect ratio.

## 4. HUD layout

Рекомендуемая иерархия:

1. health / stamina;
2. move stick;
3. attack / dodge / interact / heal;
4. pause;
5. inventory / journal / secondary surfaces.

Правила:

- health/stamina должны быть читаемыми, но компактными;
- кнопки лучше делать icon-first или short-label;
- Russian labels должны быть короткими и понятными;
- HUD не должен закрывать чекпойнт, NPC или центр боя.

## 5. Performance

- Держать карту и эффекты маленькими.
- Ограничивать количество динамических светов.
- Использовать лёгкие particles.
- Избегать expensive shaders.
- Не перекрывать экран большим числом полупрозрачных слоёв.

## 6. Memory

- Не загружать огромные atlases без нужды.
- Не держать в памяти лишние архивы.
- Не раздувать ассеты ради одного тестового экрана.

## 7. Particles

- Частицы должны быть редкими и короткими.
- Атмосфера лучше через немного ash/fog/glow, чем через десятки слоёв.
- Переизбыток частиц на mid-range Android быстро портит кадр.

## 8. Lighting

- Использовать свет экономно.
- Предпочитать fake glow sprites там, где они дешевле.
- Ограничивать число активных PointLight2D.

## 9. Aspect ratios

- Тестировать минимум на нескольких landscape пропорциях.
- Проверять 16:9, 18:9 и близкие к ним варианты.
- Не полагаться на один size preset.

## 10. Device QA

Обязательно проверять:

- управление пальцем;
- reachability кнопок;
- readability UI;
- отсутствие перекрытий;
- отсутствие shaky layout при смене кадра;
- отсутствие заметного input lag.

## 11. Android Editor limitations

- Android Editor полезен для проверки логики и layout, но не заменяет реальный девайс.
- Некоторые эффекты и производительность могут отличаться от desktop editor.
- Если layout кажется спорным, проверяется на device-sized viewport и затем на устройстве.

## 12. Export and signing

- Не коммитить keystore и exported artifacts.
- Экспортную конфигурацию документировать отдельно.
- Версию и package name держать под контролем.

## 13. Common Android mistakes

- HUD слишком велик и лезет в центр;
- кнопки слишком близко к краям;
- кнопки слишком прозрачны;
- текст слишком мелкий;
- карты слишком большие для экрана;
- персонаж и враги плохо читаются;
- слишком много декоративных слоёв без пользы;
- забытая проверка на real device.
